## 왜 코드 탐색이 중요합니까?

첫 출근한다고 상상해보십시오. 코드베이스에 파일이 수천 개입니다. 다 읽으려면 몇 주가 걸릴 것입니다. 그런데 오늘 당장 버그를 고쳐야 합니다.

이럴 때 Claude로 코드 탐색이 빛을 발합니다. 폴더를 일일이 뒤지는 대신, Claude에게 필요한 것을 찾아달라고 하면 됩니다.

**실제 상황 예시:**
- 프로젝트를 인수받았는데 어떻게 동작하는지 이해해야 함
- 어딘가에 버그가 있는데 어떤 파일을 봐야 할지 모름
- 기능을 추가하고 싶은데 참고할 비슷한 코드를 찾아야 함
- 다른 사람 코드를 리뷰하는데 맥락을 이해해야 함
- 레거시 코드를 유지보수해야 함

> 💡 **비전공자 팁**
>
> **Claude를 코드 관광 가이드라고 생각하세요:**
> - Claude 없이: 낯선 거리를 돌아다니며 식당 찾기
> - Claude와 함께: 모든 것이 어디 있는지 정확히 아는 동네 친구와 함께
>
> 코드를 직접 다 읽을 필요 없습니다.
> Claude에게 물어보면 핵심을 파악해서 설명해줍니다.

## 코드 리딩의 중요성

### 왜 코드 리딩이 중요한가?

개발자는 **코드를 작성하는 시간보다 읽는 시간이 더 많습니다**.

| 활동 | 시간 비율 |
|------|----------|
| 코드 읽기/이해하기 | 60-70% |
| 코드 작성하기 | 20-30% |
| 디버깅/테스트 | 10-20% |

코드를 빠르게 이해하는 능력이 곧 생산성입니다!

### 효과적인 코드 리딩 전략

#### 1. 하향식 접근 (Top-Down)

전체 그림부터 파악하고, 세부사항으로 들어갑니다.

```
1단계: 프로젝트 개요 파악
> 이 프로젝트가 뭐야? 한 줄로 설명해줘.

2단계: 구조 이해
> 폴더 구조 보여줘. 각 폴더 역할 설명해줘.

3단계: 진입점 찾기
> 앱이 어디서 시작해? 메인 파일이 뭐야?

4단계: 특정 기능 탐색
> 로그인 기능은 어떻게 구현되어 있어?
```

#### 2. 기능 중심 접근 (Feature-based)

관심 있는 기능을 따라가며 이해합니다.

```
> 사용자가 버튼을 클릭하면 어떤 일이 일어나?
> 데이터는 어디서 어디로 흘러가?
> 이 API 호출은 어디서 시작해서 어디서 끝나?
```

#### 3. 역추적 접근 (Backward Tracing)

결과물에서 시작해서 원인을 추적합니다.

```
> 이 화면이 어디서 렌더링돼?
> 이 에러 메시지는 어디서 나와?
> 이 데이터는 어디서 오는 거야?
```

> 🔥 **프로 팁**
>
> **코드 리딩의 80/20 법칙:**
>
> 프로젝트의 **20% 코드**가 **80% 기능**을 담당합니다.
> 그 핵심 20%를 먼저 파악하세요.
>
> 핵심을 찾는 질문:
> - "가장 중요한 파일 5개가 뭐야?"
> - "비즈니스 로직은 어디 있어?"
> - "사용자 요청은 어떻게 처리돼?"

## 검색 도구: Glob과 Grep

Claude Code는 두 가지 강력한 검색 도구를 사용합니다.

### Glob: 파일 이름으로 찾기

Glob은 **패턴**을 사용해서 파일 이름으로 검색합니다.

#### 기본 패턴

| 패턴 | 의미 | 예시 |
|------|------|------|
| `*.js` | .js로 끝나는 파일 | app.js, utils.js |
| `*.{ts,tsx}` | .ts 또는 .tsx 파일 | Button.tsx, api.ts |
| `**/*.js` | 모든 폴더의 .js 파일 | src/app.js, lib/utils.js |
| `src/**/*` | src 안의 모든 파일 | src/..., src/components/... |
| `**/test*` | test로 시작하는 파일 | test.js, testing.md |
| `**/*config*` | config 포함된 파일 | tsconfig.json, vite.config.ts |

#### Glob 사용 예시

```
# JavaScript/TypeScript 파일 찾기
> 이 프로젝트의 모든 TypeScript 파일 찾아줘

# 테스트 파일 찾기
> 테스트 파일들 어디 있어?
> .test.ts나 .spec.ts로 끝나는 파일 찾아줘

# 설정 파일 찾기
> config 들어간 파일 다 찾아줘

# 특정 폴더에서 찾기
> src/components 안의 .tsx 파일 다 찾아줘
```

> 💡 **비전공자 팁**
>
> **패턴을 외울 필요 없어요!**
>
> 자연어로 요청하면 Claude가 알아서 적절한 패턴으로 검색합니다.
> "Button 관련 파일 찾아줘"라고 하면 됩니다.

### Grep: 파일 내용에서 찾기

Grep은 파일 **내용**에서 텍스트를 검색합니다.

#### Grep 사용 예시

```
# 특정 함수 찾기
> "handleSubmit" 함수 어디에 정의되어 있어?

# 특정 텍스트 찾기
> "TODO" 주석 다 찾아줘

# 특정 패턴 찾기
> console.log 남아있는 곳 다 찾아줘

# API 엔드포인트 찾기
> "/api/" 포함된 코드 다 찾아줘

# 에러 메시지 찾기
> "사용자를 찾을 수 없습니다" 메시지 어디서 나와?
```

#### 고급 Grep 패턴

```
# 함수 정의 찾기
> "function fetchUser" 정의된 곳 찾아줘

# import 문 찾기
> useState 임포트하는 파일 다 찾아줘

# 특정 변수 사용처 찾기
> isLoading 변수 사용하는 곳 다 찾아줘

# 정규식 패턴 (고급)
> 이메일 형식 검증하는 정규식 찾아줘
```

### Glob vs Grep 비교

| 특성 | Glob | Grep |
|------|------|------|
| 검색 대상 | 파일 이름 | 파일 내용 |
| 사용 예 | "Button.tsx 찾아줘" | "handleClick 어디서 호출해?" |
| 질문 유형 | "어떤 파일이 있어?" | "이 코드가 어디 있어?" |

> ⚠️ **주의사항**
>
> **node_modules는 검색에서 제외됩니다.**
>
> 설치된 패키지 코드는 검색 결과에 포함되지 않습니다.
> 이건 의도된 동작이에요. 수천 개의 불필요한 결과가 나오는 걸 방지합니다.

package.json과 설정 파일을 분석해서 기술 스택을 알려줍니다.

```

```

## 탐색 워크플로우

새 프로젝트를 이해하는 체계적인 순서입니다.

### 5단계 탐색 워크플로우

```
┌─────────────────────────────────────────────────────────────┐
│  1단계: 프로젝트 개요                                         │
│  > 이 프로젝트가 뭐야? 간단히 설명해줘.                         │
│  > 기술 스택이 뭐야?                                          │
├─────────────────────────────────────────────────────────────┤
│  2단계: 폴더 구조                                             │
│  > 폴더 구조 보여주고 각 폴더 역할 알려줘.                       │
│  > 가장 중요한 폴더 3개가 뭐야?                                │
├─────────────────────────────────────────────────────────────┤
│  3단계: 진입점                                               │
│  > 이 앱이 어디서 시작해? 메인 파일이 뭐야?                      │
│  > package.json의 scripts 설명해줘                            │
├─────────────────────────────────────────────────────────────┤
│  4단계: 핵심 파일                                             │
│  > 가장 중요한 파일 5개 알려줘                                  │
│  > 비즈니스 로직은 어디에 있어?                                 │
├─────────────────────────────────────────────────────────────┤
│  5단계: 특정 기능                                             │
│  > [관심 있는 기능] 어떻게 구현되어 있어?                        │
│  > 관련 파일 찾아서 설명해줘.                                   │
└─────────────────────────────────────────────────────────────┘
```

### 실제 적용 예시

```
# 1단계: 개요
> 이 프로젝트가 뭐야?

Claude: 이 프로젝트는 Next.js 기반의 블로그 플랫폼입니다.
Markdown으로 글을 작성하고, 태그별 분류와 검색 기능을 지원합니다.

# 2단계: 구조
> 폴더 구조 설명해줘

Claude:
src/
├── app/          # Next.js App Router (페이지)
├── components/   # UI 컴포넌트
├── lib/          # 유틸리티 함수
├── content/      # Markdown 글 파일
└── styles/       # CSS 스타일

# 3단계: 진입점
> 메인 페이지가 어디야?

Claude: src/app/page.tsx가 메인 페이지입니다.
최근 글 목록을 보여주고 있습니다.

# 4단계: 핵심 파일
> 가장 중요한 파일 5개 알려줘

Claude:
1. src/app/page.tsx - 메인 페이지
2. src/lib/posts.ts - 글 데이터 처리
3. src/components/PostCard.tsx - 글 카드 UI
4. src/app/posts/[slug]/page.tsx - 글 상세 페이지
5. src/lib/markdown.ts - Markdown 파싱

# 5단계: 특정 기능
> 검색 기능 어떻게 구현되어 있어?

Claude: @src/app/search/page.tsx에 검색 페이지가 있습니다.
@src/lib/search.ts에서 Fuse.js를 사용해 퍼지 검색을 구현했습니다.
검색 인덱스는 빌드 타임에 생성됩니다.
```

## Plan 모드로 안전하게 탐색

파일을 바꾸지 않고 탐색만 하고 싶을 때 Plan 모드를 사용하십시오.

```
> /plan
> 이 프로젝트 분석해줘. 수정하지 말고 설명만.
```

### Plan 모드에서 할 수 있는 것

| 작업 | Plan 모드 | Normal 모드 |
|------|----------|-------------|
| 파일 읽기 | O | O |
| 구조 분석 | O | O |
| 파일 수정 | X | O |
| 명령어 실행 | X | O |
| 파일 생성 | X | O |

### Plan 모드 활용 예시

```
> /plan
> 이 프로젝트의 아키텍처 분석해줘.
> 어떤 패턴을 사용하고 있는지.
> 개선할 점이 있는지.
> 수정은 하지 말고 분석만.

Claude: [상세한 분석 제공]
아키텍처 분석:
1. 현재 패턴: Feature-based 구조
2. 상태 관리: React Query 사용
3. 스타일링: Tailwind CSS
...

개선 제안:
1. 타입 안전성 강화 필요
2. 에러 처리 일관성 부족
...

[파일 수정 없이 분석만 완료]
```

> 💡 **비전공자 팁**
>
> **Plan 모드가 유용한 상황:**
>
> - 처음 프로젝트를 분석할 때
> - 다른 사람 코드 리뷰할 때
> - 무엇을 수정할지 결정하기 전에
> - 실수로 파일을 변경하고 싶지 않을 때

## 대규모 코드베이스 탐색

프로젝트가 클 때의 전략입니다.

### 범위 제한하기

```
# 전체 프로젝트가 너무 크면
> src/auth/ 폴더만 분석해줘

# 특정 기능에 집중
> 결제 관련 코드만 찾아줘

# 특정 파일 타입만
> TypeScript 파일 중에서 "user" 들어간 것만 찾아줘
```

### 단계별 질문

```
# 전체 다 읽지 말고 점진적으로
> 먼저 폴더 구조만 보여줘.
[결과 확인]

> 그 중에서 api 폴더 자세히 볼게.
[결과 확인]

> api/users.ts 파일 설명해줘.
[결과 확인]

> 이 파일에서 createUser 함수 설명해줘.
```

### 핵심 먼저 파악

```
# 중요한 것부터
> 이 프로젝트에서 가장 핵심적인 파일 5개만 알려줘

# 변경 빈도 높은 파일
> git log 보고 가장 자주 수정된 파일 알려줘

# 의존성 많은 파일
> 다른 파일에서 가장 많이 import하는 파일 찾아줘
```

> 🔥 **프로 팁**
>
> **대규모 프로젝트 탐색 전략:**
>
> 1. **먼저 README 읽기**: 프로젝트 개요 파악
> 2. **package.json 확인**: 사용 기술 파악
> 3. **진입점 찾기**: 앱 시작 지점 파악
> 4. **관심 영역 집중**: 전체를 다 볼 필요 없음
> 5. **점진적 확장**: 필요할 때 범위 넓히기

## 특정 상황별 탐색 패턴

### 버그를 찾을 때

```
> 이 에러 메시지가 어디서 나오는 거야?
> "사용자를 찾을 수 없습니다" 메시지 검색해줘

> 이 함수 호출하는 곳 다 찾아줘
> fetchUser 함수 사용하는 파일 목록

> 이 상태가 변경되는 곳 찾아줘
> isAuthenticated 값이 바뀌는 곳 다 찾아줘
```

### 기능을 추가할 때

```
> 비슷한 기능 어디 있어?

## 🚨 흔한 에러와 해결법

### 에러 1: "파일을 찾을 수 없음"

**상황:** Claude가 분명 있는 파일을 못 찾음

**원인:**
- 올바른 디렉토리에 있지 않음
- 파일 이름이 예상과 다름

**해결:**
```
> 지금 어느 폴더에 있어?
> 현재 폴더에서 Button 들어간 파일 찾아줘
> 대소문자 구분 없이 찾아줘
```

### 에러 2: "결과가 너무 많음"

**상황:** 검색 결과가 100개 이상

**원인:**
- 검색 범위가 너무 넓음

**해결:**
```
# 범위 좁히기
> components 폴더 안의 Button 파일만 찾아줘
> .ts 파일에서만 찾아줘
> src 폴더 안에서만 찾아줘
```

### 에러 3: "프로젝트에 접근할 수 없음"

**상황:** Claude가 프로젝트 파일을 못 읽음

**원인:**
- 프로젝트 폴더 안에서 Claude Code를 실행하지 않음

**해결:**
```bash
cd /path/to/project
claude
```

### 에러 4: "탐색이 너무 오래 걸림"

**상황:** 응답이 매우 느림

**원인:**
- 프로젝트가 너무 큼
- 요청 범위가 너무 넓음

**해결:**
```
# "모든 것" 대신 구체적으로
> 전체 프로젝트 말고 src 폴더만 분석해줘
> 모든 파일 말고 가장 중요한 5개만 알려줘
```

## 🆘 안 되면?

| 증상 | 해결 방법 |
|------|----------|
| 파일을 못 찾음 | 폴더 위치 확인, 더 구체적인 이름 사용 |
| 결과가 너무 많음 | 검색 범위 좁히기, 폴더 제한 |
| 너무 느림 | 범위 제한, "모든 것" 대신 "핵심 5개" |
| 구조가 복잡함 | 하향식 접근, 한 영역씩 탐색 |
| 코드가 이해 안 됨 | "비전공자도 이해할 수 있게 설명해줘" |

## 📚 관련 용어 사전

| 용어 | 설명 |
|------|------|
| **Glob** | 파일 이름 패턴 매칭 도구 |
| **Grep** | 파일 내용 텍스트 검색 도구 |
| **진입점 (Entry Point)** | 앱이 시작되는 첫 파일 |
| **의존성 (Dependency)** | 다른 모듈을 참조하는 관계 |
| **정규식 (RegEx)** | 텍스트 패턴 매칭 문법 |
| **하향식 (Top-Down)** | 전체에서 세부로 접근하는 방식 |
| **코드 리딩** | 코드를 읽고 이해하는 활동 |
