### 왜 아키텍처를 알아야 하나요?

자동차를 운전할 때 엔진 구조를 몰라도 됩니다. 하지만 엔진이 어떻게 작동하는지 알면:
- 이상한 소리가 날 때 뭐가 문제인지 짐작할 수 있고
- 연비를 더 좋게 만드는 운전법을 이해할 수 있고
- 고장 났을 때 정비사에게 정확히 설명할 수 있습니다

Claude Code도 마찬가지입니다. 내부 작동 원리를 알면:
- 왜 어떤 요청은 빠르고 어떤 요청은 느린지 알 수 있습니다
- 비용을 절약하는 요청 방법을 이해할 수 있습니다
- 문제가 생겼을 때 스스로 해결할 수 있습니다

> 💡 **비전공자 팁**: 이 챕터가 조금 기술적으로 느껴질 수 있어요. 하지만 걱정 마세요! 모든 개념을 쉬운 비유로 설명할 거예요. 100% 이해 못 해도 괜찮습니다. 대략적인 원리만 알아도 Claude Code를 더 잘 쓸 수 있어요.

## 왜 필요합니까?

**실제 상황**: "이 프로젝트의 모든 TypeScript 파일을 찾아서 에러를 고쳐줘"라고 요청했는데 시간도 오래 걸리고 비용도 많이 나왔습니다. 왜 그럴까요? 내부 작동 원리를 모르기 때문입니다.

아키텍처를 이해하면:
- 어떤 요청은 빠르고 어떤 요청은 느린지 알 수 있습니다
- 문제가 생겼을 때 디버깅할 수 있습니다
- 효율적인 요청으로 비용을 절약할 수 있습니다

### 쉬운 비유: 레스토랑 주방

Claude Code를 레스토랑에서 주문하는 것처럼 생각해 보시기 바랍니다:

```
당신 (손님) --> 웨이터 (Claude Code CLI) --> 주방 (Anthropic API)
                                                    |
                                                    v
                                              셰프가 요리 준비
                                                    |
                                                    v
                              웨이터 <-- 음식 가져옴 --> 당신이 식사
```

- **웨이터** (CLI)가 주문을 받아서 주방과 소통합니다
- **주방** (API)에서 실제 요리(생각)가 이루어집니다
- 주방에 갈 때마다 시간이 걸립니다 (API 왕복)
- "뭐든 맛있는 걸로 해주세요"라고 하면 웨이터가 여러 번 왕복해야 합니다

## 왜 아키텍처를 알아야 합니까?

Claude Code를 "마법의 도구"로 생각하면 한계가 있습니다. 작동 원리를 이해하면:

- **더 정확한 요청**: 어떤 도구가 사용될지 예측 가능
- **문제 해결**: 왜 안 되는지 스스로 파악 가능
- **효율적 사용**: 불필요한 작업 줄이기

## Claude Code의 전체 구조

### 한 눈에 보기

```
┌─────────────────────────────────────────────────────────────────┐
│                        여러분 (터미널)                           │
└─────────────────────────────────────────────────────────────────┘
                               │
                               ▼
┌─────────────────────────────────────────────────────────────────┐
│                       Claude Code CLI                            │
│  ┌───────────────┐  ┌───────────────┐  ┌───────────────────┐    │
│  │ 입력 처리     │  │ 도구 실행 엔진 │  │  출력 렌더러       │    │
│  └───────────────┘  └───────────────┘  └───────────────────┘    │
└─────────────────────────────────────────────────────────────────┘
                               │
                               ▼
┌─────────────────────────────────────────────────────────────────┐
│                    Anthropic API (Claude)                        │
│               모델: opus / sonnet / haiku 선택 가능              │
└─────────────────────────────────────────────────────────────────┘
```

**핵심 포인트**: Claude는 여러분의 컴퓨터에서 직접 실행되지 않습니다. API를 통해 대화하고, 도구 실행만 로컬에서 일어납니다.

### API가 무엇입니까? (초보자용)

**API** (Application Programming Interface)는 두 프로그램 사이의 메신저와 같습니다. Claude Code를 사용할 때:

1. 여러분의 컴퓨터가 Anthropic 서버로 메시지를 보냅니다
2. Claude (Anthropic 서버에서)가 요청을 처리합니다
3. 응답이 여러분의 컴퓨터로 돌아옵니다
4. Claude가 무언가를 해야 하면 (파일 읽기, 명령 실행 등) 그것은 여러분의 컴퓨터에서 일어납니다

**이것이 왜 중요합니까?**
- Claude는 직접 요청하기 전까지 여러분의 파일을 볼 수 없습니다
- Claude가 질문할 때마다 = 또 한 번의 왕복 = 시간과 비용 추가
- 처음부터 구체적으로 요청하면 왕복 횟수를 줄일 수 있습니다

### API 통신 사이클

요청을 하면 어떤 일이 일어나는지 봅시다:

```
 [1] 여러분의 요청                    [2] API로 전송
      │                                 │
      ▼                                 ▼
┌──────────┐    메시지 + 도구 정의    ┌─────────────┐
│ CLI      │ ─────────────────────▶ │ Anthropic   │
│ Client   │                        │ API         │
└──────────┘                        └─────────────┘
      ▲                                 │
      │     응답 (텍스트 + 도구 호출)     │
      └─────────────────────────────────┘
                    [3]

 [4] 로컬에서 도구 실행               [5] 결과 다시 API로
      │                                 │
      ▼                                 ▼
┌──────────────────┐              ┌─────────────┐
│ Bash, Read 등    │ ───────────▶ │ Claude가    │
│ 실행             │              │ 다음 판단   │
└──────────────────┘              └─────────────┘
```

### 이것을 알면 무엇이 좋습니까?

```
> 이 폴더의 모든 파일 보여줘
```

이 요청을 하면:
1. Claude가 `Glob` 도구로 파일 목록 조회 → API 왕복 1번
2. 결과를 받아서 다음 행동 결정 → API 왕복 2번
3. 필요하면 `Read`로 파일 내용 확인 → API 왕복 3번

**한 번의 요청 = 여러 번의 API 왕복**이 될 수 있습니다. 그래서 구체적으로 요청하면 왕복 횟수가 줄어들고 더 빠릅니다.

## 도구 시스템

Claude Code는 17개의 내장 도구를 사용합니다.

### 전체 도구 목록

| 카테고리 | 도구 | 설명 |
|---------|------|------|
| **파일 작업** | `Read` | 파일 읽기 (이미지, PDF, Jupyter 노트북 지원) |
| | `Write` | 파일 생성/덮어쓰기 |
| | `Edit` | 문자열 치환 기반 파일 편집 |
| | `Glob` | 패턴으로 파일 검색 (예: `**/*.ts`) |
| | `Grep` | 내용 검색 (ripgrep 기반) |
| **실행** | `Bash` | 셸 명령 실행 |
| | `KillShell` | 백그라운드 셸 종료 |
| **에이전트** | `Task` | 서브에이전트 생성 및 실행 |
| | `TaskOutput` | 백그라운드 작업 결과 조회 |
| **웹** | `WebFetch` | URL 내용 가져오기 (HTML→Markdown) |
| | `WebSearch` | 웹 검색 |
| **계획** | `EnterPlanMode` | 계획 모드 진입 |
| | `ExitPlanMode` | 계획 모드 종료 및 승인 요청 |
| **사용자 상호작용** | `AskUserQuestion` | 사용자에게 질문 (객관식) |
| | `TodoWrite` | 작업 목록 관리 |
| **기타** | `Skill` | 스킬 실행 (/commit, /review-pr 등) |
| | `NotebookEdit` | Jupyter 노트북 셀 편집 |

| 도구 | 제한 | 값 |
|------|------|-----|
| **Bash** | 기본 타임아웃 | 2분 |

| | 출력 제한 | 30,000자 |
| **Read** | 기본 라인 수 | 2,000 라인 |
| | 라인 길이 | 2,000자 |

**요청 방식에 따라 사용되는 도구가 다릅니다:**

```
# Glob 사용 (빠름)

```

Claude Code의 전용 도구가 있는 작업은 그 도구를 쓰게 요청하는 게 효율적입니다.

### AskUserQuestion 도구

이 도구는 작업 실행 중에 Claude가 여러분에게 질문할 수 있게 합니다.

**언제 사용되나요?**
- 모호한 요구사항 명확히 하기
- 여러 접근 방식 중 선택하기
- 구현 세부사항에 대한 선호도 확인

**예시 상호작용:**

```
Claude: 인증에 대해 확인이 필요합니다.

┌─ 인증 방식 ───────────────────────────────────────────┐
│ 어떤 인증 방식을 사용할까요?                           │
│                                                        │
│ ○ JWT (권장)                                           │
│   상태 비저장, API에 적합                              │
│                                                        │
│ ○ 세션 기반                                            │
│   전통적인 방식, 서버측 저장                           │
│                                                        │
│ ○ OAuth                                                │
│   소셜 로그인 (Google, GitHub)                         │
└────────────────────────────────────────────────────────┘
```

**왜 중요합니까?**
추측하거나 가정하는 대신 Claude가 직접 물어볼 수 있습니다. 이렇게 하면 실제 요구사항에 맞는 더 좋은 결과를 얻을 수 있습니다.

## 서브에이전트 시스템

복잡한 작업은 여러 "서브에이전트"로 나뉘어 실행됩니다.

### 서브에이전트 아키텍처

```
                          ┌─────────────────┐
                          │   메인 에이전트  │
                          │   (여러분과 대화) │
                          └────────┬────────┘
                                   │
                    ┌──────────────┼──────────────┐
                    │              │              │
                    ▼              ▼              ▼
         ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
         │   Task 1     │ │   Task 2     │ │   Task 3     │
         │  (탐색)      │ │  (분석)      │ │  (실행)      │
         └──────────────┘ └──────────────┘ └──────────────┘
```

### 서브에이전트 타입

| 타입 | 용도 | 사용 가능 도구 |
|------|------|---------------|
| `Explore` | 코드베이스 탐색, 파일 검색 | Read, Glob, Grep (Edit/Write 제외) |
| `Plan` | 구현 계획 수립, 아키텍처 설계 | Read, Glob, Grep (Edit/Write 제외) |
| `Bash` | Git 작업, 명령 실행 | Bash만 |
| `general-purpose` | 복잡한 멀티스텝 작업 | 모든 도구 (*) |
| `claude-code-guide` | Claude Code 사용법 안내 | Glob, Grep, Read, WebFetch, WebSearch |
| `statusline-setup` | 상태라인 설정 | Read, Edit |

### 병렬 vs 순차 실행

```
병렬 실행 (독립적인 작업들):
─────────────────────────────────────────────
    ┌──────────┐     ┌──────────┐     ┌──────────┐
    │ Agent A  │     │ Agent B  │     │ Agent C  │
    │ (검색1)  │     │ (검색2)  │     │ (검색3)  │
    └────┬─────┘     └────┬─────┘     └────┬─────┘
         │                │                │
         └────────────────┼────────────────┘
                          │
                          ▼
                   ┌──────────────┐
                   │ 결과 통합     │
                   └──────────────┘

순차 실행 (의존적인 작업들):
─────────────────────────────────────────────
    ┌──────────┐     ┌──────────┐     ┌──────────┐
    │ Agent A  │ ──▶ │ Agent B  │ ──▶ │ Agent C  │
    │ (분석)   │     │ (A결과   │     │ (B결과   │
    │          │     │  사용)   │     │  사용)   │
    └──────────┘     └──────────┘     └──────────┘
```

### 이것을 알면 무엇이 좋습니까?

**큰 작업을 요청하면 자동으로 분리됩니다:**

```
> 이 프로젝트 전체 분석하고 리팩토링 계획 세워줘
```

내부적으로:
1. `Explore` 에이전트가 프로젝트 구조 파악
2. `Plan` 에이전트가 계획 수립
3. 메인 에이전트가 결과 종합

**작업을 미리 나눠서 요청하면 더 정확합니다:**

```
# 한 번에 다 하기 (에이전트가 알아서 나눔)
> 프로젝트 분석하고 리팩토링해줘

# 단계별로 나누기 (더 정확한 제어)
> 먼저 프로젝트 구조만 분석해줘
> (결과 확인 후)
> 이 부분 리팩토링 계획 세워줘
> (계획 확인 후)
> 좋아, 실행해줘
```

## 모델 선택

Claude Code는 여러 모델을 지원합니다.

| 모델 | 특징 | 언제 사용 |
|------|------|----------|
| **Opus** | 가장 똑똑함, 비용 높음 | 복잡한 설계, 어려운 버그 |
| **Sonnet** | 균형잡힌 성능 | 일반적인 작업 (기본값) |
| **Haiku** | 빠르고 저렴 | 간단한 작업, 반복 작업 |

### 실용적인 전략

```
# 계획 단계: 똑똑한 모델로
> /model opus
> 이 시스템 어떻게 설계하면 좋을까?

# 구현 단계: 빠른 모델로
> /model sonnet
> 위 계획대로 구현해줘
```

**비용 vs 품질 트레이드오프**를 상황에 맞게 조절할 수 있습니다.

## 컨텍스트 관리

### 대화가 길어지면 품질이 떨어집니다

Claude는 한 번에 처리할 수 있는 정보량에 한계가 있습니다. 대화가 길어지면:
- 초반 내용을 잘 기억 못함
- 전체 맥락 파악이 흐려짐
- 응답 품질 저하

### 해결 방법

**1. /compact로 압축**
```
> /compact
```
지금까지의 대화를 요약해서 토큰을 절약합니다.

**2. 주제별로 대화 분리**
```
# 한 대화에서 다 하기 (맥락이 섞임)
> 인증 기능 만들어줘
> DB 스키마도 수정해줘
> 프론트엔드 폼도 만들어줘

# 주제별로 나누기 (더 집중)
> /clear
> 인증 기능에 집중하자. 백엔드 API 먼저 만들어줘.
```

**3. CLAUDE.md 활용**

CLAUDE.md는 `/clear` 후에도 유지됩니다. 프로젝트 규칙을 여기에 적어두면 대화를 새로 시작해도 컨텍스트가 보존됩니다.

## 권한과 보안

### 보안 정책 흐름

```
  사용자 요청
       │
       ▼
┌──────────────┐    아니오     ┌──────────────┐
│ 위험한 작업? │ ────────────▶ │ 바로 실행    │
└──────┬───────┘               └──────────────┘
       │ 예
       ▼
┌──────────────┐    거부       ┌──────────────┐
│ 사용자 승인  │ ────────────▶ │ 작업 취소    │
│ 요청         │               └──────────────┘
└──────┬───────┘
       │ 승인
       ▼
┌──────────────┐
│ 샌드박스에서 │
│ 실행         │
└──────────────┘
```

### 왜 승인을 요청합니까?

Claude Code는 위험할 수 있는 작업 전에 승인을 요청합니다:
- 파일 수정/삭제
- 명령어 실행
- 외부 API 호출

### 자동 승인 설정

자주 쓰는 안전한 도구는 자동 승인할 수 있습니다:

```json
// settings.json
{
  "permissions": {
    "autoApprove": ["Read", "Glob", "Grep"]
  }
}
```

**주의**: `Edit`, `Write`, `Bash`는 신중하게 설정하시기 바랍니다.

### 샌드박스 모드

Bash 명령은 기본적으로 샌드박스에서 실행됩니다. 시스템을 보호하면서 작업할 수 있습니다.

## 실용적인 팁

### 1. 도구를 명시적으로 언급

```
# 암시적 (Claude가 추론해야 함)
> 이 프로젝트에서 에러 처리 어떻게 되어있어?

# 명시적 (더 빠름)
> grep으로 "catch" 키워드 검색해서 에러 처리 패턴 보여줘
```

### 2. 한 번에 하나씩

```
# 여러 작업 한꺼번에 (혼란 가능)
> 파일 만들고, 테스트 작성하고, 커밋해줘

# 단계별로 (더 정확)
> 파일 먼저 만들어줘
> (확인 후) 테스트 작성해줘
> (확인 후) 커밋해줘
```

### 3. 결과 확인 후 진행

Claude가 파일을 수정했다면 결과를 확인한 후 다음 단계로 진행하시기 바랍니다. 중간에 잘못된 것이 있으면 바로 수정할 수 있습니다.

## 따라해보세요

도구 시스템을 이해하기 위한 간단한 실습입니다:

### 실습 1: 도구 관찰하기

```
> 이 폴더에 파일이 몇 개야?
```

Claude의 응답을 관찰하시기 바랍니다. `Glob`을 사용했습니까, `Bash`를 사용했습니까? 결과는 같지만 효율성이 다릅니다.

### 실습 2: 접근 방식 비교하기

두 가지 요청을 시도하고 차이를 느껴보시기 바랍니다:

```
# 접근 방식 A - 모호함
> 이 프로젝트에 대해 알려줘

# 접근 방식 B - 구체적
> package.json을 읽고 이 프로젝트가 어떤 의존성을 사용하는지 알려줘
```

접근 방식 B가 더 빠릅니다. Claude가 어떤 도구(Read)를 사용할지 정확히 알기 때문입니다.

### 실습 3: 왕복 횟수 세기

```
> 이 프로젝트에서 모든 TODO 주석을 찾아서 목록으로 보여줘
```

몇 번의 도구 호출이 일어나는지 관찰하시기 바랍니다. 각각이 API 왕복입니다.

## 안 되면?

### 문제: Claude가 느리거나 멈춘 것 같아요

**가능한 원인:**
1. 요청이 많은 도구 호출(왕복)을 필요로 함
2. 매우 큰 파일을 처리 중
3. API와의 네트워크 문제

**해결 방법:**
- 큰 요청을 작은 단계로 나누시기 바랍니다
- 원하는 것을 구체적으로 말하시기 바랍니다
- 인터넷 연결을 확인하시기 바랍니다
- 맥락이 혼란스러우면 `/clear`로 새로 시작하시기 바랍니다

### 문제: Claude가 잘못된 도구를 사용합니다

**예시:** 텍스트 검색을 요청했는데, Claude가 내장 `Grep` 도구 대신 `Bash`로 `grep`을 실행합니다.

**해결 방법:** 필요할 때 도구를 명시적으로 언급하시기 바랍니다:
```
# 이것 대신
> 이 프로젝트에서 "TODO"를 검색해줘

# 이렇게 시도
> grep 도구로 모든 파일에서 "TODO"를 검색해줘
```

### 문제: 권한 거부 에러

**원인:** Claude Code는 위험한 작업에 대해 권한이 필요합니다.

**해결 방법:** `y`를 눌러 승인하거나, 안전한 도구에 대해 설정에서 자동 승인을 구성하시기 바랍니다.

## 자주 하는 실수

1. **한 번에 너무 많이 요청하기**
   - 나쁨: "이 프로젝트 분석하고, 모든 버그 고치고, 테스트 추가하고, 배포해줘"
   - 좋음: "먼저 프로젝트 구조를 분석해줘"

2. **위치를 구체적으로 말하지 않기**
   - 나쁨: "설정 파일 보여줘"
   - 좋음: "tsconfig.json 파일 보여줘"

3. **컨텍스트 제한을 잊기**
   - 긴 대화는 컨텍스트를 잃습니다
   - 주기적으로 `/compact`나 `/clear`를 사용하시기 바랍니다

4. **도구 출력을 무시하기**
   - 다음으로 넘어가기 전에 Claude가 실제로 무엇을 했는지 항상 확인하시기 바랍니다
   - 실수를 일찍 발견할 수 있습니다

5. **비용을 이해하지 못하기**
   - 왕복 횟수가 많을수록 = API 호출이 많을수록 = 비용 증가
   - 구체적인 요청이 더 저렴합니다
